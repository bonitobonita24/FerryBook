Source: Airbnb from getdesign.md
Generated by: Planning Assistant Phase 2.8 (V31) · last revised after Batch 13 (Emergency Cancellation + Travel Credit + Discount-ID Verification)
Install command: `npx getdesign@latest add airbnb` (run from project root after Phase 3 generates inputs.yml)

## Visual Theme

Travel-marketplace aesthetic anchored by a warm coral primary color that signals friendly action without aggression. Photography-driven hero areas dominate the customer-facing surface; admin and back-office areas use the same color tokens but lean on whitespace and typography rather than imagery. Generous breathing room, soft shadows on cards, and pill-shaped interactive elements give the interface a consumer-app feel that distances F and S Marine Transport Inc. from the typical clunky enterprise booking portals (2GO Online, PHBus) that dominate the Philippine ferry market. Trust signals — MARINA license badges, total-passengers-served counters, safety record — are styled as muted secondary elements so they support without competing.

## Voice & Tone

| Surface            | Tone                                                                     |
|--------------------|--------------------------------------------------------------------------|
| Customer-facing    | Friendly, concise, second-person ("Pick your date"); never corporate     |
| Hero copy          | Aspirational but plain-spoken — "Book your trip to Lubang Island in just a minute." Never make claims the product can't deliver (e.g. don't say "ferry crossing in 60 seconds" when the crossing is 4 hours — refer to *booking* time, not *travel* time) |
| Confirmation messages | Warm and reassuring ("You're booked! See you at the pier.")           |
| Error messages     | Direct and helpful — what failed + what to do next; never blame the user |
| Recovery flows (refund / reschedule / no-show / emergency) | Empathetic but action-oriented; acknowledge the disruption without dwelling ("Your sailing was cancelled" → "Pick how you'd like to recover — your booking value is fully protected") |
| Seat selection | Concrete and visual ("Pick your seats" → "Pick N more seats" → "Continue · A05-C, A05-D, A05-E →"). Never abstract ("Select seating preferences"). The button label always shows actual seat IDs when complete so the customer can verify before paying |
| Account-owner explainer | Plain-spoken about the silent link — "This name plus your contact phone identifies your account. If a phone number you enter below is already linked to an F&S Marine account, this booking will attach to that account automatically — no separate sign-in needed." Never use security/identity jargon ("authentication", "credentials"); never imply the customer must do something extra |
| Discount-ID reminders | Frank but kind ("Bring the right ID for each passenger" — never "WARNING:"); explain consequences without scolding |
| Counter staff prompts | Procedural; use imperative voice ("Ask passenger for: Senior Citizen ID"); brief because staff are mid-task |
| Boarding Officer / MC-180 | Formal, regulatory — this is a legal document. "Master/Captain" not "Captain". Cite MARINA / PCG license numbers verbatim |
| Admin UI           | Neutral and professional; assumes domain literacy (no over-explaining)   |
| Emergency cancellation broadcast preview | Sober and unambiguous — "This action cannot be undone." Always specify customer count, max liability, and the recovery options being offered |
| Email templates    | Personal opener ("Hi Maria,") — never "Dear Customer"                    |
| SMS templates      | Maximum 160 chars; lead with brand name ("F&S Marine:"); include reference + key facts only; never marketing copy in an OTP or refund SMS |

## Mockup-only conventions (do NOT carry into production)

The deployed `docs/vercel_mockup/` Vite app uses several shortcuts that exist *because it is a mockup*, and they should never bleed into the production Next.js build:

| Convention | Where | Why it's mockup-only |
|------------|-------|----------------------|
| Tailwind via CDN (`<script src="https://cdn.tailwindcss.com">`) | `docs/vercel_mockup/index.html` | Faster iteration; production uses Tailwind as a build-time dep with full purge |
| Single-file 16k-line `MOCKUP.jsx` containing all 40+ screens | `docs/vercel_mockup/src/MOCKUP.jsx` | Easy to share + diff in one file; production splits per-route under `apps/web/src/app/**` |
| Tiny path-match router in `main.jsx` (regex on `window.location.pathname`) | `docs/vercel_mockup/src/main.jsx` | Avoids a router dependency for two routes; production uses Next.js App Router |
| `📐 Mockup demo` / `📐 Mockup control` dashed-border strips | inside many screens | Hint to viewers that the surrounding control is fake (e.g. "use code 123456") — strip these before any production port |
| Demo OTP codes (`123456` for booking, `654321` for login) | hard-coded in OTP screens | Production must validate against real UniSMS attempts |
| **No nested phone frame in PWA previews** — the Counter and Boarding PWA preview cards render full-width inside the mockup's device frame, NOT inside a second phone silhouette | `docs/vercel_mockup/src/MOCKUP.jsx` "Native Mobile Apps" section | Avoids a frame-in-frame visual; production won't have a device-frame wrapper at all |
| `pcOnlyScreens = ['staffWalkin']` viewport gate | `FandSMarineMockup` shell at the bottom of `MOCKUP.jsx` | Mockup hides the mobile/tablet preview for the walk-in screen so reviewers see it the way the staff PC at the counter would; production renders it Mobile First on a terminal tablet (see PRODUCT.md → Mobile Needs row 40) — **this is a mockup-display choice, not a product decision** |
| Mock occupied-seats computed pseudo-randomly per class (`(r * (c+1) * seed) % 5 === 0`) | walk-in seat grid + customer seat selection in `MOCKUP.jsx` | Production reads `Seat.status` from PostgreSQL |
| `cdn.tailwindcss.com` is blocked in some corp networks | `docs/vercel_mockup/index.html` | If the mockup fails to style on a reviewer's machine, that's why — production has no such dep |

## Phase 4 Implementation Note (for Claude Code)

After Phase 3 generates `inputs.yml`, run from project root:

```bash
npx getdesign@latest add airbnb
```

This installs the base official Airbnb DESIGN.md tokens into the project. This file (`docs/DESIGN.md`) is the Phase 2.8 extraction *plus* the locked architectural colors (class, port, role, passenger-type, recovery-mode, status badge) that emerged through Batches 7-13 — the npx-installed file does NOT include those, so this document is authoritative for anything beyond the base palette.

If the two ever conflict on the base palette (background, foreground, primary, ink, success, warning, destructive, border, muted), prefer the npx-installed `DESIGN.md` at project root for runtime token values. For everything else — class colors, port colors, role colors, status badge map, passenger-type pills, recovery-mode color conventions, semantic alert tones, reference formats, component patterns specific to F and S Marine, voice & tone — this file is canonical and must be honored.

Locked tokens that Phase 4 must NOT reassign:
- Open Air class = `#1E40AF` foreground / `#DBEAFE` background
- Aircon class = `#FF385C` (= primary) foreground / `#FFE5E9` background
- VIP class = `#A16207` foreground / `#FEF3C7` background
- Boarding Officer role = `#7C3AED` foreground / `#EDE9FE` background
- Travel credit theme = `#7C3AED` action / `#5B21B6` ink / `#EDE9FE` tint
- Emergency cancellation = `#E63946` (= destructive) → `#7F1D1D` gradient
- BAT-NAS port = coral (primary), BAT-CAL port = amber (`#A16207`), MIN-TIL = gray
- Status badge color map (10 statuses) — verbatim per the table above
- All reference-format prefixes (`FSM-`, `RR-`, `NSR-`, `NSB-`, `EMC-`, `EMR-`, `CRD-`)
- Seat capacity contract: Open Air = 80 (10×8, IDs `O01-A` through `O10-H`); Aircon = 50 (10×5, IDs `A01-A` through `A10-E`); VIP = 12 (3×4, IDs `V01-A` through `V03-D`). Aisle convention: Aircon has aisle between columns C and D; VIP has aisle between columns B and C; Open Air has no aisle. The Seat entity in PRODUCT.md derives from these locked counts — changing them requires re-issuing all future Voyage seat rows and a vessel-config migration

These weren't free decisions — they encode domain meaning the rest of the app depends on (e.g. a passenger seeing a purple background instantly knows it's about credits, not about a no-show; an admin seeing destructive red on a banner knows the action is irreversible). Reassigning them will silently break the UX even though the code still compiles.
